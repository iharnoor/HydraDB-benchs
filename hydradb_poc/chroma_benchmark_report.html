The Core Problem: Cosine Similarity ≠ Understanding

Pure vector databases find chunks that are semantically similar to your query. But similarity is not understanding. When you ask "What trips were NOT in Florida?", the word "trip" + "Florida" pulls Florida trips to the top. When you ask about the "day after" an event, embeddings have no concept of time. These aren't edge cases — they're fundamental limitations.

Negation Blindness
"NOT birthday" returns birthday chunks first
No Temporal Awareness
"Day after X" returns X, not what follows
No Entity Direction
Can't distinguish "A did X for B" from "B did X for A"
No Geographic Filtering
"Outside the US" returns American events
Aggregation Ceiling
Top-k returns 5 of 10+ relevant chunks
Semantic Traps
"Engaged/married?" returns wedding attendance

🚫 Negation — "Vector DBs Can't NOT"
When your query contains a word you want to EXCLUDE, cosine similarity treats it as a positive match signal. Asking "not birthday" makes birthday chunks score HIGHER, not lower.
Negation
Negation — Not a Birthday
Both Fail
Question
What did Harnoor and Katie celebrate together that was NOT a birthday?
Expected: Navratri, July 4th, Valentine's Day, Thanksgiving, Christmas, New Year's, wedding
ChromaDB 44.8ms
#1 Retrieved chunk:
Katie's Birthday: Harnoor celebrated Katie's birthday...
HydraDB 507ms
#1 Retrieved chunk:
Harnoor celebrated Katie's birthday with a white layered cake...
Both systems return birthday chunks first. This is the clearest proof that retrieval systems don't understand negation. The word "birthday" in the query boosts birthday chunks — even though the user explicitly asked for NOT birthdays. An LLM downstream would need to filter, but the retrieval layer already failed.

⏰ Temporal Adjacency — "Embeddings Have No Clock"
Vector embeddings encode meaning, not time. They can't answer "what happened next?" or "put these in order."

➡ Entity Direction — "WHO Did WHAT to WHOM"
Embeddings flatten "Katie gave Harnoor a gift" and "Harnoor gave Katie a gift" into nearly identical vectors. Direction is invisible to cosine similarity.

📊 Aggregation — "Top-K Can't Count"
When 10+ chunks are relevant but you only retrieve 5, you structurally cannot give a complete answer.
Aggregation
Exhaustive Listing of Celebrations
Both Fail
Question
List every holiday or celebration Harnoor and Katie shared together
Expected: 10+ events — both birthdays, NYE x2, Valentine's, Navratri, July 4th, Thanksgiving, Christmas, wedding
ChromaDB 43.3ms
Returns 5 chunks: Christmas, Thanksgiving, Katie's birthday, Navratri, Harnoor's birthday. Misses: NYE x2, Valentine's, July 4th, wedding.
HydraDB 500ms
Returns 5 chunks: Christmas, summary doc, Navratri, July 4th, Harnoor's birthday. Misses: NYE x2, Valentine's, Thanksgiving, Katie's birthday, wedding.
Both systems are structurally limited to top-5 retrieval. With 10+ celebrations spread across 19 chunks, neither can return a complete list. This is a fundamental limitation of retrieval-based systems — aggregation queries need either full-scan access or a graph that can traverse "celebration" nodes.

Key Takeaways

What this benchmark proves about the limits of pure vector search.

Vector Search Is Not Memory

Cosine similarity finds related documents, not correct answers. "Related" breaks down with negation, temporal queries, entity direction, and geographic filtering. 4 of 19 tests failed for BOTH systems, proving these are structural limitations of retrieval, not just Chroma.

Hybrid Retrieval Matters

HydraDB won 7 of 19 tests by combining vector similarity with graph edges (temporal adjacency, entity relationships) and BM25 (keyword precision). The day-after-Princeton test is the clearest win — graph edges connect July 3 → July 4 automatically.

Vector Search Still Has Strengths

For direct fact retrieval ("What car?", "When did they meet?", "What did she wear?"), pure vector search works well. 6 tests tied because the answer was a simple semantic match. ChromaDB is also 14x faster (local vs cloud).

LLM Layer Can't Fix Bad Retrieval

Adding GPT/Gemini/Claude on top doesn't fix retrieval failures. If the wedding attendance chunk is #1 for "are they married?", the LLM will likely hallucinate a wedding. If all top-5 chunks are Florida trips, no LLM can find the Princeton answer. Garbage in, garbage out.
